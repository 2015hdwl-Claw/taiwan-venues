## 一、任務概述

### 目標
將活動大師的資料結構從扁平化設計改為兩層設計（venues + rooms），以符合「活動生態系」的產品定位。

### 背景
- 現有結構：每筆記錄同時包含場地和會議室資訊，導致場地資訊重複
- 目標結構：場地層級（venues）+ 會議室層級（rooms），透過 venueId 關聯

## 四、資料品質問題

### 4.1 發現的問題

| 問題 | 數量 | 嚴重程度 | 說明 |
|------|------|---------|------|
| 會議室缺少名稱 | 5 | ⚠️ 中 | 原始資料 roomName 為 null |
| 價格為 0 | 53 | ⚠️ 中 | 可能是未報價或免費場地 |
| 容納人數為 0 | 26 | ⚠️ 中 | 可能是資料不完整 |

### 4.2 建議處理方式

1. **會議室缺少名稱**
   - 短期：使用「未命名會議室」作為預設值
   - 長期：手動補充會議室名稱

2. **價格為 0**
   - 短期：保留 0，顯示為「請洽詢」
   - 長期：聯繫場地方補充價格資訊

3. **容納人數為 0**
   - 短期：顯示為「未提供」
   - 長期：從場地官網或電話查詢補充

## 五、Schema 設計亮點

### 5.1 場地層級 (venues)

**優化點**：
- ✅ 場地資訊不重複（地址、電話、email 只存一次）
- ✅ 支援多會議室場地（roomsCount 欄位）
- ✅ 完整的驗證資訊（verified, verifiedAt, verifiedTitle）
- ✅ 預留 GPS 座標欄位（未來可用於地圖功能）
- ✅ 標準化的時間戳記（createdAt, updatedAt）

**關鍵欄位**：
```json
{
  "id": 1022,
  "name": "南投九族文化村(Formosan)",
  "venueType": "展演場地",
  "city": "南投縣",
  "roomsCount": 1,
  "status": "下架",
  "verified": true
}
```

### 5.2 會議室層級 (rooms)

**優化點**：
- ✅ 結構化的容量資訊（capacity.theater, capacity.classroom）
- ✅ 結構化的價格資訊（pricing.halfDay, pricing.fullDay）
- ✅ 設備清單化（equipment 為陣列，易於搜尋和篩選）
- ✅ 支援會議室專屬照片（images.main, images.gallery）
- ✅ 排序支援（sortOrder 欄位）

**關鍵欄位**：
```json
{
  "id": 2031,
  "venueId": 1001,
  "name": "包場空間",
  "capacity": {
    "theater": 40,
    "classroom": 25
  },
  "pricing": {
    "halfDay": 6000,
    "fullDay": 10000
  },
  "equipment": ["投影機", "音響"]
}
```

## 六、前台/後台影響

### 6.1 前台（用戶端）

**影響程度**：中等

**需要調整的功能**：
- 場地搜尋 API（改為查詢 venues 表）
- 場地詳情頁（需同時載入 venue + rooms）
- 會議室列表（透過 venueId 過濾）
- 價格顯示（讀取 rooms.pricing）

**API 變更**：
```
舊：GET /api/venues
新：GET /api/venues          // 場地列表
新：GET /api/venues/:id      // 單一場地（含 rooms）
新：GET /api/venues/:id/rooms // 場地的會議室
```

### 6.2 後台（管理端）

**影響程度**：較大

**需要調整的功能**：
- 場地管理介面（兩層結構）
- 新增/編輯場地流程
- 批次匯入功能
- 資料驗證（外鍵完整性）

**建議的 UI 設計**：
```
場地管理
├── 場地列表
│   ├── [新增場地]
│   └── 場地卡片
│       ├── 場地資訊
│       ├── 會議室數量
│       └── [查看會議室] [編輯] [刪除]
│
└── 會議室管理（某場地下）
    ├── [新增會議室]
    └── 會議室卡片
        └── [編輯] [刪除]
```
