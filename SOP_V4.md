## 1. 目標與範圍

### 目標
確保台灣會議場地資料庫的**完整性**、**準確性**與**一致性**。

### 核心原則
> **資料對了才有價值，資料錯了一文不值。**
> **一次做對，不要重來。**
> **所有網址都要記錄，下次更新不用重新找。**
> **照片優先使用官方來源，避免使用第三方。** ⭐ 新增
> **同一品牌/機構的場地要完整記錄。** ⭐ 新增

## 2. 驗證流程

### Phase 1: 官網驗證
```
1. 開啟官網 URL
2. 檢查 HTTP 狀態碼
3. 檢查網頁標題是否匹配
4. 記錄錯誤類型（DNS、SSL、Timeout 等）
```

### Phase 2: 會議室頁面尋找
```
1. 搜尋關鍵字：會議、宴會、場地、租借、活動
2. 找到會議室相關連結
3. 開啟會議室頁面
4. 記錄 meetingPageUrl
```

### Phase 3: 會議室完整清單 ⭐ 強制執行
```
1. 在會議室頁面尋找所有會議室名稱
2. 記錄每個會議室的：
   - 名稱（roomName）
   - 容納人數（maxCapacity）
   - 價格（如有）
   - 照片（如有）
3. 統計會議室總數（roomsCount）
4. 比對資料庫現有記錄數量
5. ⭐ 強制：為每個會議室創建獨立記錄
6. ⭐ 強制：驗證會議室數量與官網一致
7. 記錄 venueListUrl（會議室清單頁面）

失敗條件：
- 會議室數量與官網不一致 → 待修
- roomsCount 未記錄 → 待修
- 缺少會議室記錄 → 待修
```

### Phase 4: 照片抓取 ⭐ 重要更新
```
1. 優先從官方網站抓取照片
   - 場地外觀、大廳、建築照
   - 會議室內部照
2. 禁止使用第三方來源（Wikipedia、新聞網站等）
3. 記錄照片來源頁面（必須是官方網站）
4. 同時更新 venueMainImageUrl 和 images.main
```

### Phase 5: 品牌/機構場地檢查 ⭐ 新增
```
1. 識別品牌/機構名稱（如集思、晶華、圓山等）
2. 搜尋該品牌/機構的所有場地
3. 確認資料庫是否已完整記錄
4. 補充缺失的場地記錄
```

### Phase 6: 資料一致性檢查 ⭐ 新增
```
1. 檢查 venueMainImageUrl 和 images.main 是否一致
2. 檢查同一場地的多筆記錄是否共享 venueMainImageUrl
3. 檢查照片來源是否為官方網站
4. 修正不一致的資料
```

### Phase 7: 狀態判定
```
1. 根據驗證結果判定狀態
2. 更新資料庫欄位
3. 標記問題類型
```

## 3. 網址資訊記錄規範

### 3.1 ⭐ 三大網址強制要求（V4.7 新增）

> **每個會議室資料都必須要有至少 3 個網址，目的是讓錯誤發生時可以找到原因。**

| # | 網址類型 | 欄位名稱 | 說明 | 必要性 |
|---|---------|---------|------|--------|
| 1 | **場地官網網址** | `url` | 場地的官方網站首頁 | ✅ 強制 |
| 2 | **主照片網址** | `images.main` | 場地主要照片（官網來源） | ✅ 強制 |
| 3 | **會議室網址** | `venueListUrl` / `meetingPageUrl` | 會議室清單頁面 | ✅ 強制 |
| 4 | **照片來源網址** | `images.source` | 照片來源頁面 URL | ✅ 強制 |

### 3.2 三大網址驗證流程

```
1. 開啟 url（場地官網）
   - 記錄 verifiedTitle（網頁標題）
   - 記錄 lastVerified（驗證時間戳）

2. 抓取 images.main（主照片）
   - 必須來自官網
   - 記錄 images.source（來源頁面）
   - 記錄 images.verifiedAt（驗證時間戳）

3. 記錄 venueListUrl（會議室頁面）
   - 會議室清單頁面 URL
   - 或 meetingPageUrl
```

### 3.3 錯誤追蹤機制（V4.7 新增）

> **目的：當錯誤發生時，可以追溯到是哪次更新造成的。**

每個場地資料必須包含以下追蹤欄位：

```json
{
  "lastVerified": "2026-03-03T14:20:00Z",
  "verifiedTitle": "網頁標題",
  "verifiedAt": "2026-03-03T14:20:00Z",
  "lastUpdated": "2026-03-03T14:20:00Z",
  "updateReason": "初次驗證 / 照片更新 / 會議室數量修正",
  "updateSource": "手動驗證 / Gemini 協助 / 自動爬蟲"
}
```

### 3.4 必須記錄的網址

每個場地必須記錄以下網址資訊：

| 網址類型 | 欄位名稱 | 說明 |
|---------|---------|------|
| **主網頁** | `url` | 場地的官方網站首頁 |
| **會議室清單頁** | `venueListUrl` | 會議室清單頁面 |
| **場地主照片** | `venueMainImageUrl` | 場地主要照片（所有會議室共用） |
| **會議室照片** | `roomImageUrl` | 會議室照片（每個會議室不同） |

## 5. 照片來源規範 ⭐ 重要

### 5.1 照片來源優先順序

| 優先順序 | 來源 | 說明 |
|----------|------|------|
| 1 | 官方網站 | ✅ 最優先 |
| 2 | 場地 Facebook 專頁 | ✅ 可接受 |
| 3 | 場地 Instagram | ✅ 可接受 |
| 4 | Google Maps | ⚠️ 僅在無其他來源時使用 |
| 5 | Wikipedia | ❌ **禁止使用** |
| 6 | 新聞網站 | ❌ **禁止使用** |

### 5.2 照片欄位一致性

| 欄位 | 說明 | 一致性要求 |
|------|------|-----------|
| `venueMainImageUrl` | 場地主照片 | 必須與 `images.main` 相同 |
| `images.main` | 主照片 | 必須與 `venueMainImageUrl` 相同 |
| `images.source` | 照片來源 | 必須是官方網站 URL |

### 5.3 照片更新流程

```
1. 從官方網站抓取照片 URL
2. 同時更新 venueMainImageUrl 和 images.main
3. 記錄 images.source 為官方網站 URL
4. 標記 images.verified = true
```

## 7. 資料一致性檢查 ⭐ 新增

### 7.1 一致性檢查清單

| 檢查項目 | 說明 | 修正方式 |
|---------|------|---------|
| venueMainImageUrl = images.main | 兩者應該相同 | 更新為相同的官方照片 URL |
| 照片來源 | 必須是官方網站 | 重新從官方網站抓取 |
| 同一場地多筆記錄 | 應共享 venueMainImageUrl | 統一使用同一個 URL |

### 7.2 一致性檢查腳本

```javascript
// 檢查 venueMainImageUrl 和 images.main 是否一致
const inconsistent = venues.filter(v => 
  v.venueMainImageUrl && v.images && v.images.main &&
  v.venueMainImageUrl !== v.images.main
);

// 檢查照片來源是否為 Wikipedia
const wikipedia = venues.filter(v => 
  v.images && v.images.main && v.images.main.includes('wikipedia')
);

// 檢查缺少 venueMainImageUrl 的上架場地
const missing = venues.filter(v => 
  v.status === '上架' && !v.venueMainImageUrl
);
```

| 狀態 | 值 | 說明 |

| **上架** | `上架` | 營運中，venueListUrl 完整，照片為官方來源 |

## 8.5 失敗重試機制 ⭐ V4.6 新增

### 8.5.1 二次驗證流程

```
1. 對所有標記為「DNS 失敗」的場地進行二次測試
2. 使用 curl 直接測試 HTTP 狀態碼
3. 區分錯誤類型：
   - DNS 解析失敗 → 域名不存在，curl 完全無回應
   - 連線超時 → 有 DNS 但連線逾時
   - HTTP 403 → 反爬蟲機制
   - HTTP 404 → 頁面不存在
   - HTTP 301/302 → 重定向（仍可訪問）
   - 內容提取失敗 → HTTP 200 但無法解析內容
4. 記錄實際錯誤類型，而非統一歸類
5. 更正誤判的場地狀態
```

### 8.5.2 curl 測試指令

```bash
# 測試官網可達性
curl -sI --connect-timeout 10 "$url" | head -1

# 結果判斷
- 無回應 → ❌ DNS 解析失敗
- HTTP 200 → ✅ 正常
- HTTP 301/302 → ✅ 重定向（仍可訪問）
- HTTP 403 → ⚠️ 反爬蟲（非 DNS 問題）
- HTTP 404 → ❌ 頁面不存在（非 DNS 問題）
```

### 8.5.3 錯誤分類精確化

| 原分類 | 正確分類 | 說明 |
|--------|----------|------|
| DNS 解析失敗 | DNS 解析失敗 | 域名真的不存在 |
| DNS 解析失敗 | 連線超時 | DNS 正常但連線逾時 |
| DNS 解析失敗 | HTTP 403 | 反爬蟲機制 |
| DNS 解析失敗 | HTTP 200 | 誤判，應為正常 |
| DNS 解析失敗 | HTTP 301/302 | 誤判，應為正常 |
| ID 不存在 | ID 存在 | Sub-agent 誤判，需用 id-query.js 驗證 |

### 8.5.4 ID 查詢驗證

```bash
# 查詢單一 ID
node id-query.js query 1433

# 批次查詢
node id-query.js batch 1433 1436 1439

# 搜尋重複記錄
node id-query.js duplicates "集思台大會議中心"
```

**重要**：使用 `==` 進行寬鬆比對，避免類型不匹配問題

## 10. ⚠️ 歷史教訓（V4.7 新增）

### 10.1 2026-03-03 TICC 誤下架事件

#### 問題描述
- **受影響場地**：TICC（45 筆）、台北喜來登（6 筆）、南港展覽館（5 筆）、政大公企中心（7 筆）
- **總計**：63 筆資料被誤下架
- **照片被覆蓋**：TICC 原本 6 張官網照片被覆蓋成 hero.jpg + 2 張 default.jpg

#### 根本原因

| 問題 | 根本原因 | 修正方式 |
|------|----------|----------|
| **roomsCount 是 undefined** | 資料結構中沒有這個欄位，被誤判為「無會議室」 | 下架判斷應排除 undefined |
| **照片被覆蓋** | 驗證腳本重新抓取照片，覆蓋了原本正確的照片 | 照片更新應檢查是否已有高品質照片 |
| **無法追溯原因** | 沒有記錄更新原因和時間戳 | 加入錯誤追蹤機制 |

#### 防止再次發生

```javascript
// ❌ 錯誤的下架判斷
if (v.roomsCount === 0) {
  v.status = '下架';  // 會把 undefined 也當成 0
}

// ✅ 正確的下架判斷
if (v.roomsCount !== undefined && v.roomsCount === 0) {
  v.status = '下架';  // 只處理明確為 0 的情況
}

// ✅ 更安全的判斷
if (v.roomsCount === 0 && v.verified === true) {
  v.status = '下架';  // 需要已驗證才能下架
}
```
